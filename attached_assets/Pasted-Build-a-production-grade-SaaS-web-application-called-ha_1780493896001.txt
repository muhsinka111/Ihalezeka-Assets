Build a production-grade SaaS web application called İhaleZeka — an AI-powered Turkish public procurement (kamu ihale) intelligence platform. It is the Turkish-market adaptation of cleat.ai (CLEATUS), which serves the US government-contracting market. Our market is the Turkish EKAP system (Elektronik Kamu Alımları Platformu) and tenders governed by Law No. 4734 (Kamu İhale Kanunu).

The entire UI must be in Turkish. I am attaching HTML mockups of every screen — match their layout, structure, and component design closely. They are the source of truth for look & feel.

TECH STACK

	•	Frontend: Next.js 14 (App Router) + TypeScript + Tailwind CSS
	•	UI: shadcn/ui, Tabler Icons (@tabler/icons-react), Sora (headings) + Inter (body) fonts
	•	State/Data: TanStack Query for server state; Zustand for light client state
	•	Backend: Next.js API routes (or a separate Express service if cleaner)
	•	Database: PostgreSQL via Prisma ORM (Replit DB or Neon)

	•	AI layer: Provider-agnostic — user supplies their OWN API key (OpenAI / Anthropic / Gemini). Store keys encrypted (AES-256), never log them.
	•	Payments: Stripe (Trial / Essential / Pro / Enterprise + per-seat)

GLOBAL: Sliding AI Panel

Right-side sliding panel (“İhaleZeka’ya Sor”, BETA) on every page via the top-bar “AI Asistan” button — mirrors CLEATUS’s “Ask CLEATUS”. Shows scored recommendations (e.g. 98/100), agency logos, CPV codes, status badges (PİPELINE’DA / YENİ / İZLENİYOR), action buttons (Pipeline’a ekle / Aç / Çıkar). Powered by the user’s connected AI key. Streaming responses.

DATA MODEL (Prisma)

	•	Organization → Business (1-n) → User (n, roles: owner/admin/member)
	•	CompanyProfile (wizard data: nace[], cpv[], experienceCeiling, certifications[], personnel{}, strategy{}, automation{})
	•	Tender (İKN, title, agencyName, agencyLogoUrl, type, method, estimatedValue, deadline, cpvCodes[], rawDocsUrls[], sourceSystem)
	•	Match (tender ↔ profile, fitScore 0-100, reasoning, status)
	•	PipelineItem (tender, stage enum, notes)
	•	Proposal (tender, contentJson, status, versions)
	•	Document (business, folder, name, fileUrl, validUntil, status)
	•	ApiKey (business, provider enum, encryptedKey, lastUsedAt)
	•	Subscription (Stripe: plan, seats, status)

AI MATCHING LOGIC (core value)

Given a CompanyProfile and a Tender, compute a 0-100 fit score by checking: CPV/NACE overlap, experience ceiling vs estimated value (≥50% rule), financial criteria, personnel requirements, geography preference. Return score + Turkish reasoning + pros/risks. Do the hard rule checks in code; use the user’s LLM only for the reasoning text.

DATA SOURCE (most critical — ASK ME)

The product’s lifeblood is tender data from EKAP. EKAP has no simple public bulk API like SAM.gov. Do not invent a data source. Build against a clean TenderRepository interface with a mock/seed adapter for now (realistic Turkish seed data — see mockups). Make swapping in a real adapter trivial later. Flag this in the README.

LEGAL CONSTRAINT

Do NOT build fully-automatic e-signature submission to EKAP. E-teklif requires a qualified electronic signature (NES); auto-signing on a user’s behalf is a serious legal/security risk. Flow: AI prepares → user reviews → user signs & submits. Keep the “otomatik e-imza” toggle present but disabled with an explanatory note.

SECURITY

	•	Encrypt stored API keys (AES-256), never log/expose them.
	•	Multi-tenant isolation: every query scoped to the user’s organization/business.
	•	KVKK (Turkish GDPR) compliant, privacy-by-default.
